--------------------------------------------------------------------------------
SECTION A — Overall development approach (Data Engineer perspective)
--------------------------------------------------------------------------------

EN:
From a data-engineering viewpoint, this project is built as a **contract-first,
row-second** pipeline: define the logical data model, respect dependencies, generate
rows, then materialize to sinks. The intent mirrors typical DE work: schema and
lineage before bulk data movement.

1) **Schema as the source of truth**  
   Tables, columns, primary/foreign keys, and types are represented in a small
   internal model (`DataSchema`). They can come from (a) built-in scenario templates
   aligned to business cases (CRM / trading / credit risk), (b) SQLite-style DDL
   parsing, or (c) JSON schema with FK hints. This parallels maintaining dbt/SQL
   models or API contracts before datasets are produced.

2) **Dependency-aware ordering**  
   A topological sort over FK relationships defines both **generation order** and
   **insert order** into SQLite so parent keys exist before children—conceptually
   similar to ordering tasks in an ETL DAG.

3) **Synthesis strategy (rules + optional ML)**  
   The default path is **rule-based**: reproducible RNG, skewed numerics for money-like
   fields, weighted categoricals, and JSON built from stable key catalogs. **Faker**
   optionally improves human-readable fields (names, timestamps, etc.) without
   breaking keys. **SDV HMA** is optional for learning multi-table joint behavior
   when training-like tables are available—similar to choosing between heuristic
   QA data and model-based synthesis for richer distributions.

4) **Semi-structured columns**  
   JSON/XML-like content is generated as structured strings with predictable paths,
   avoiding “random JSON soup” that breaks parsers and tests—aligned with how DE
   teams keep event payloads usable for analytics.

5) **Physicalization & scale**  
   Exporters write SQLite (batched inserts), Parquet per table, and optional Delta.
   The design separates “how rows are made” from “where they land,” as in separating
   transformation code from warehouse/lake writers.

6) **Verification**  
   Automated tests cover parsing, FK-safe ordering, and (optionally) SDV smoke tests—
   comparable to CI checks on schema drift and data tests before promoting a pipeline.

中文（数据工程师视角的整体开发思路）：
本项目按**先契约、后数据、再落地**的思路搭建，与常见数据工程流程一致：先有清晰 schema 与依赖，再生成数据，最后写入存储。

1）**Schema 为单一事实来源**  
   用内部模型表达多表、主外键与类型；来源可以是（a）与业务场景对齐的内置模板（CRM/交易/信贷风险），（b）解析 SQLite 风格 DDL，或（c）带 FK 元数据的 JSON。类似在 dbt/SQL 或契约里先定表结构，再产出数据集。

2）**依赖驱动的顺序**  
   对外键关系做拓扑排序，既决定**生成顺序**，也决定**向 SQLite 插入顺序**，保证父键先于子键存在；思想上接近 ETL DAG 中任务拓扑。

3）**合成策略：规则为主，可选 ML**  
   默认路径为**规则合成**：可复现随机种子、金额类偏斜分布、类别加权、JSON 用稳定键集合生成。可选 **Faker** 提升姓名字段等可读性，但不替代主外键逻辑。若有训练样本，可选 **SDV HMA** 学习多表联合分布——对应“启发式造数”与“基于样本的拟合造数”的分工。

4）**半结构化字段**  
   JSON 等列生成带稳定结构与业务语义的字符串，避免完全随机键值导致解析与测试失效；与生产里希望日志/事件 JSON 可被稳定解析的思路一致。

5）**落地与扩展**  
   导出层支持 SQLite（批量插入）、按表 Parquet、可选 Delta；**生成逻辑**与**写入目标**解耦，便于替换存储或接入更大规模引擎（如 Spark）。

6）**验证**  
   自动化测试覆盖解析、外键顺序与（可选）SDV 烟测；对应流水线发布前的 schema/数据质量门禁思路。

SECTION B — 5 Likely Data-Engineer Questions (with EN / 中文 answers)

--------------------------------------------------------------------------------
Q1. How do you ensure referential integrity across multiple tables when generating
    synthetic data, and what can break it in production pipelines?
--------------------------------------------------------------------------------

EN:
Referential integrity is enforced by generating tables in topological order (parents
before children) and sampling foreign-key values only from the already-generated
primary-key pool of the referenced table. Exporters should insert rows in the same
order so SQLite FK constraints succeed. In production, integrity breaks if generation
order is wrong, if you parallelize per table without coordinating PK pools, if you
truncate/reload tables out of order, or if the schema drifts from what the generator
assumes (missing FK metadata, type mismatches). Validation via PRAGMA foreign_key_check
(or equivalent checks in other engines) after load is a good guardrail.

中文：
参照完整性通过两件事保证：一是按拓扑序先父表后子表生成；二是外键列只从已生成父表主键集合中抽样。导出时也应按同样顺序插入，SQLite 的外键约束才能通过。生产上若生成顺序错误、并行生成各表却不同步 PK 池、按错误顺序截断/重载表，或 schema 与生成器假设不一致（缺 FK 元数据、类型不一致），都会破坏完整性。加载后用 PRAGMA foreign_key_check（或其它引擎的等价校验）做校验是常见防线。

--------------------------------------------------------------------------------
Q2. What are the trade-offs between rule-based generation (distributions + Faker)
    and model-based approaches such as SDV / CTGAN for financial synthetic data?
--------------------------------------------------------------------------------

EN:
Rule-based methods are fast, reproducible with seeds, and need no training data;
they excel at constraints and FK logic but may not match real joint distributions
or rare tail events. Model-based methods can better mimic empirical correlations and
multivariate patterns when you have representative training data, but they are
heavier, more sensitive to data quality and volume, and may leak sensitive patterns
if training data is not properly governed. Hybrid patterns are common: rules for
keys and legal ranges, models for dense numeric/categorical blocks, and always
separate PII handling (masking, synthesis policies).

中文：
规则生成（分布 + Faker）速度快、可种子复现、无需训练数据，擅长约束与 FK，但未必复现真实联合分布与尾部风险。基于模型的方法（SDV/CTGAN 等）在有代表性训练数据时能更好拟合相关性与多变量形态，但更重、对数据量与质量敏感，且若治理不当可能“拟合”敏感模式。工程上常见混合：主键与合法区间用规则，稠密数值/类别用模型，并对 PII 单独策略（脱敏、合成策略）。

--------------------------------------------------------------------------------
Q3. How should semi-structured fields (JSON) be handled so they remain useful for
    analytics and testing without blowing up schema complexity?
--------------------------------------------------------------------------------

EN:
Treat JSON as a first-class column type in the logical model, but persist as TEXT
(or native JSON where supported). Generate nested objects with stable keys and
realistic value domains (e.g. tier, channel, MCC) so parsers and BI tools can
project paths. Avoid fully random key soup; prefer a small catalog of keys and
optional arrays. For heavy analytics, consider extracting hot keys into relational
columns (or materialized views) while keeping the full JSON for audit-like use cases.

中文：
在逻辑模型里把 JSON 当作一等类型，落库可用 TEXT（或引擎原生 JSON）。生成时保持键集合稳定、取值域合理（如 tier、channel、MCC），便于下游解析与投影。避免键名完全随机；优先固定少量键与可选数组。若分析很重，可把高频字段抽到关系列或物化视图，全文 JSON 留给审计类场景。

--------------------------------------------------------------------------------
Q4. What scalability and performance issues appear when generating large datasets,
    and how would you address them?
--------------------------------------------------------------------------------

EN:
Bottlenecks include Python object overhead per row, single-threaded generation,
SQLite write throughput, and Parquet file size/memory when building huge DataFrames.
Mitigations: batch inserts (already used), chunked Parquet writes, vectorized or
columnar generation where possible, writing partitioned Parquet by table/date,
offloading to Spark/Dask for TB-scale, and separating “schema validation” from
“bulk generation” jobs. For Delta, compaction and partition strategy matter for
downstream query cost.

中文：
瓶颈包括：逐行 Python 对象开销、单线程生成、SQLite 写入吞吐、以及一次性拼巨型 DataFrame 时的内存与 Parquet 体积。对策包括：批量插入（已有）、分块写 Parquet、能向量化的列用向量化生成、按表/日期分区写出、TB 级用 Spark/Dask，并把 schema 校验与大批量生成分离。若用 Delta，压缩与分区策略会影响下游查询成本。

--------------------------------------------------------------------------------
Q5. How do you validate that synthetic data is “good enough” for downstream use
    (ML, dashboards, contract testing) without ground-truth production data?
--------------------------------------------------------------------------------

EN:
Use layered checks: (1) schema and constraint validation (types, PK/FK, NOT NULL);
(2) distribution sanity (ranges, category frequencies vs business rules); (3)
relationship checks (child counts per parent within expected ratios); (4) for ML,
simple hold-out behavior tests on pipelines (schema compatibility, no silent NaN
explosion); (5) privacy-oriented checks if training on real data (no direct re-identification
of rare combinations). Document assumptions explicitly—synthetic data is rarely a
perfect substitute for production statistics.

中文：
分层验证：（1）schema 与约束（类型、PK/FK、非空）；（2）分布合理性（取值范围、类别占比是否符合业务规则）；（3）关系合理性（子表相对父表的条数比例是否在预期内）；（4）对 ML/流水线做兼容性烟测（schema 对齐、无静默 NaN 爆炸等）；（5）若用真实数据训练合成模型，还需隐私侧检查（避免稀有组合可重识别）。并在文档中写清假设——合成数据很少能完全替代生产统计特性。
